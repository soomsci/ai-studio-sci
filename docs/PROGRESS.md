```
1회차: [0] 978-459   [1] 117-880
2회차: [0] 117-880   [1] 978-459
```

즉 `found[0]`은 **실행할 때마다 다른 센서를 가리킨다.** 6모둠이 동시에 수집기를 켜면 어느 노트북이 어느 센서를 잡을지 예측할 수 없다. 추측이 아니라 확인된 사실이다.

**센서 번호 형식** (세션 F가 알아야 할 것) — 기기 이름은 `Temperature 117-880>18` 형태이고, `connect_by_id()`에 넘길 6자리 번호는 `117-880`이다. **뒤의 `>18`을 잘라내야 한다.** (`name.split()` 후 `">"` 앞부분만 취하면 됨. 이 처리를 빠뜨리면 번호 추출이 조용히 실패한다 — 세션 G도 처음에 이걸로 틀렸다.)

**다중 센서 지원 → 진행하기로 결정(2026-07-20).** 2대 동시 연결이 되므로 한 노트북이 여러 센서를 같은 조건에서 동시에 잴 수 있다. 사용자가 "온도 센서 2대로 물·식용유를 동시에 비교"하는 수업을 원해 **프롬프트 ⑥으로 세션 F에 요청**. §5 스키마는 그대로 두고(측정 1회 = 센서 1개), 센서마다 `datasets` 문서를 하나씩 만드는 방식이다. 실험 2(조도와 CO₂를 같은 조건에서 비교)에도 그대로 도움이 된다.

**부수 확인 사항**

- pasco가 보고하는 단위는 `'DegC'`이지만, `sensor.py`는 `PASCO_SENSORS`에 하드코딩한 `'℃'`를 쓰고 pasco 단위를 참조하지 않는다. **SPEC §5의 `unit` 규약을 지키는 올바른 처리다** — 고칠 필요 없음.
- pasco의 검색은 **기기 이름에 센서 종류 문자열이 포함되는지** 보는 단순 매칭이다(`'Temperature' in "Temperature 117-880>18"`). 세션 F의 `PASCO_SENSORS` 키가 이 매칭에 그대로 쓰이므로 키 이름을 임의로 바꾸면 검색이 깨진다.
- `_compatible_devices` 목록에 `CO2`, `Light`, `Temperature`, `Voltage`, `Current`가 모두 있어, 세션 F가 `PASCO_SENSORS`에 넣은 5종은 검색 단계까지는 문제없다.

  넘겨야 할 번호는 "117-880"이다. 뒤의 ">18"을 잘라내야 한다.
  (name.split() 후 ">" 앞부분만 취하면 된다. 이 처리를 빠뜨리면
   번호 추출이 조용히 실패하고 엉뚱한 경로를 탄다 — 세션 G도 처음에 틀렸다.)

- main.py의 /api/connect도 이미 deviceId를 받아 connect(device_id)로
  넘기고 있다. 그런데 templates/index.html에 입력칸이 없어서
  항상 None으로 호출되고, 결국 found[0] 경로를 탄다.

할 일:
1. templates/index.html에 "센서 번호" 입력칸을 추가해라.

3. 번호를 비워두면 지금처럼 아무 센서나 잡는데, 그게 왜 위험한지
   화면에 짧게 알려주는 게 좋다. 다만 번호를 못 찾는 상황도 있으니
   빈칸 허용 자체를 막지는 마라.
4. 번호가 틀렸을 때(그 번호 센서를 못 찾음) 나오는 오류 메시지를
   한국어로 다듬어라. 지금은 pasco 예외가 그대로 올라올 수 있다.

5. collector/sensor_check.py를 만들어라. 센서가 안 잡힐 때 교사가

   출력은 전부 한국어로, 다음을 순서대로 확인해 알려주게 해라:
   - 블루투스로 주변 PASCO 센서 검색 (몇 개 찾았는지, 각각 이름)
   - 하나도 없으면: 전원이 켜져 있는지, 다른 앱이 연결을 잡고 있진

   - 연결 후 get_sensor_list() / get_measurement_list() / 단위 출력
   - PASCO_SENSORS에 적힌 measurement 추측값과 실제 목록을 대조해
     일치 여부를 명시적으로 알려주기 (CO2 센서가 들어왔을 때
