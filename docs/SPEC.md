- 이 문서가 수정되면 **부록 B 변경 이력**에 기록합니다. 이미 진행 중인 세션이 있다면, 그 세션에 변경분을 알려 반영시킵니다.

## 1. 무엇을 만드는가

### 1.1 배경

한 학기 동안 3개의 과학 탐구 프로젝트를 진행합니다. 학생들은 PASCO 무선 센서로 데이터를 직접 수집하고, 분석하고, 결론을 도출해 산출물을 만듭니다.

| | 실험 | 차시 | 관련 성취기준 | 핵심 데이터 |
|---|---|---|---|---|
| 1 | 우리 교실 최적 환기 주기 찾기 | 5 | [9과01-01] [9과04-01] | 시간에 따른 CO₂ 농도 |
| 2 | 교실 식물 최적 조건 찾기 | 5 | [9과12-01] [9과12-02] | 조건별 CO₂ 변화량 |
| 3 | 운동과 우리 몸 | 7 | [9과13-02~05] | 심박수·폐활량·날숨 CO₂ |

학기 전체를 관통하는 질문: **"CO₂는 어디서 와서 어디로 갈까?"**
교실에 쌓이고(실험1) → 식물이 흡수하고(실험2) → 우리 몸이 만든다(실험3).

### 1.2 이 앱의 목적

**단순한 그래프 도구가 아닙니다.** 학생이 데이터에서 스스로 결론을 이끌어내도록 안내하는 **탐구 안내 도구**입니다.

그래프를 그려주는 것까지가 절반이고, 나머지 절반은 "이 그래프에서 무엇을 읽어야 하는가"를 단계적으로 묻는 것입니다. §8과 §10이 그 부분입니다.

### 1.3 사용자

| 사용자 | 인원 | 기기 | 하는 일 |
|---|---|---|---|
| 학생 | 약 30명 / 6모둠 | 모둠당 윈도우 노트북 1대 | 측정, 분석, 결론 작성 |
| 교사 | 1명 | 노트북 | 전체 현황 확인, 학급 종합, 데이터 관리 |

## 2. 기술 스택

### 2.1 확정 사항

| 영역 | 선택 | 비고 |
|---|---|---|
| 프론트엔드 | **순수 HTML + CSS + JavaScript** | 빌드 도구·프레임워크 사용 금지 |
| 차트 | **Chart.js** (CDN) | 별도 라이브러리 추가 금지 |
| 백엔드 | **Firebase** (Firestore / Auth / Hosting) | |
| 수집기 | **Python 3.11** + `pasco` + `firebase-admin` | PyInstaller로 .exe 배포 |

### 2.2 왜 프레임워크를 쓰지 않는가

유지보수 주체가 교사 1인이고, 학기마다 내용을 직접 수정해야 합니다. `npm install` 없이 파일을 열어 고칠 수 있어야 합니다. React·Vue·Svelte·TypeScript·번들러 도입은 금지합니다.

### 2.3 브라우저

Chrome 최신 버전 기준. 다른 브라우저 대응은 하지 않습니다.

### 2.4 저장소·비밀정보 관리

Firebase 설정값(API 키 등)은 `public/js/firebase-config.js`에 두고 `.gitignore`로 제외합니다(v1.6). 저장소에는 빈 본보기 `firebase-config.example.js`만 둡니다. 세부 규칙은 `CLAUDE.md` 절대 규칙 7을 따릅니다.

참고: Firebase 웹 API 키는 본래 비밀값이 아니며(브라우저에 그대로 노출됨), 실제 보안 경계는 §5.4 보안 규칙과 인증입니다. 다만 남용 방지를 위해 **Google Cloud Console에서 HTTP 리퍼러 제한**을 걸어 둡니다(2026-07-20 적용 완료).

### 2.5 배포 (v1.9)

Firebase Hosting으로 배포합니다. 배포에 필요한 파일은 `firebase.json`, `.firebaserc` 두 개이며 **세션 A가 담당**합니다.

- **Hosting 루트는 `public/`** 입니다. `collector/`·`docs/`·`mock` 원본 등은 배포되지 않습니다.
- **단일 페이지 앱(SPA) 재작성 규칙을 넣지 마십시오.** 이 앱은 `join.html`·`index.html`·`teacher.html` **세 개의 실제 페이지**로 나뉘어 있어(§6.1), 모든 주소를 `index.html`로 돌리면 입장 화면과 교사 화면이 열리지 않습니다.
- `firebase.json`에 `firestore.rules` 경로를 함께 지정해 `firebase deploy --only firestore:rules`로 규칙을 배포할 수 있게 합니다.
- **`firebase-config.js`는 `.gitignore` 대상이지만 배포에는 반드시 포함되어야 합니다.** Hosting은 디스크에 있는 `public/` 내용을 그대로 올리므로 로컬에 파일이 있으면 배포됩니다. 다만 **저장소를 새로 내려받은 컴퓨터에서 배포하면 이 파일이 없어 앱이 조용히 "연습 모드"로 배포됩니다.** 배포 전 파일 존재를 확인하십시오.

**CLI 설치에 대하여** — `CLAUDE.md` 기술 규칙의 "`npm install` 하지 않는다"는 **앱이 실행에 쓰는 라이브러리**를 두고 한 말입니다. 배포 도구인 `firebase-tools`를 전역 설치(`npm install -g firebase-tools`)하는 것은 이 규칙과 무관하며 필요합니다. 앱 코드는 여전히 순수 HTML+CSS+JS이고 빌드 단계가 없습니다.

## 3. 절대 규칙

모든 세션이 지킵니다.

1. **자기 담당 파일 밖은 수정하지 않는다.** 필요하면 사용자에게 요청한다.
2. **§5 데이터 스키마를 임의로 변경하지 않는다.**
3. **학생 실명을 저장하지 않는다.** 모둠 단위 익명 라벨만 사용한다(`4모둠-A`).
4. **개인 건강정보(심박수·폐활량)는 개인 식별자와 결합하지 않는다.**
5. `localStorage`는 학급 코드·모둠 번호·화면 설정에만 사용한다. 측정 데이터는 Firestore에 저장한다.
6. **모든 UI 텍스트는 한국어**로, 중학교 2학년이 읽는 문장으로 쓴다. 전문 용어를 쓸 때는 괄호로 풀어 쓴다.
7. 파일 하나가 400줄을 넘으면 사용자에게 알리고 분리를 제안한다.
8. 주석은 한국어로 단다.
9. Firebase API 키 등 설정값을 공개 저장소에 커밋하지 않는다.

## 5. 데이터 계약 (Firestore 스키마)

**이 절이 세션 간 가장 중요한 약속입니다. 모든 세션이 읽습니다.**

### 5.1 컬렉션 구조

```
joinCodes/{joinCode}                        ← 입장 코드 → 학급 찾기 (v1.7)
  └─ classId       : string   해당 학급 문서 아이디

classes/{classId}
  ├─ name          : string   "2학년 3반"
  ├─ joinCode      : string   "3B-2026"   학생 입장 코드
  ├─ teacherUid    : string
  ├─ activeExp     : number   1|2|3|4  현재 진행 중인 실험
  ├─ visibleExps   : array    [1,2,3]  학생 화면에 보일 실험 탭 (v2.0)
  └─ createdAt     : timestamp

  ├─ name          : string   "4모둠"

  ├─ expNo         : number   1|2|3|4   (4 = 비열, v2.0)

  ├─ points        : array    [{ t: number, v: number }]   t=경과 초, v=측정값
  ├─ events        : array    [{ t: number, label: string }]  "창문 열기" 등

  ├─ chartOptions  : object   기준선 값, 축 설정 등

  ├─ conclusion    : string   최종 결론
  ├─ annotations   : array    [{ t: number, label: string }]  학생이 분석 중 그래프에 단 사건 메모 (v2.4)
```

### 5.2 반드시 지킬 것

- **측정 1회 = `datasets` 문서 1개.** 데이터 포인트마다 문서를 만들지 않습니다. 읽기 횟수가 폭증해 Firebase 무료 한도를 소진합니다.
- `points` 배열이 커질 수 있으므로 **문서 1MB 제한**을 넘지 않게 합니다. 측정 간격 10초 × 45분 = 270개 정도가 정상 범위입니다. 5,000개를 넘으면 경고를 표시합니다.
- `t`는 **측정 시작으로부터 경과 초**입니다. 절대 시각이 아닙니다.
- 학생은 `analyses`를 자유롭게 수정하지만 `datasets`는 자기 모둠 것만 수정합니다.
- **존재하지 않는 학급에도 쓰기가 됩니다 — 주의** (v2.2). Firestore는 부모 문서가 없어도 하위 컬렉션에 문서를 만들 수 있습니다. 즉 `classes/{없는아이디}/datasets`에 저장이 되고, 오류도 나지 않습니다. 게다가 이렇게 생긴 문서는 삭제 규칙의 `isTeacherOf` 검사가 부모 문서를 못 찾아 **앱에서 지울 수 없습니다**(관리자 권한 필요). 따라서 **쓰는 쪽에서 학급이 실제로 있는지 먼저 확인**해야 합니다.
- **학생이 분석 중 다는 사건 메모는 `analyses.annotations`에 저장합니다** (v2.4, 절대 규칙 6). 수집기가 측정 중 기록한 `datasets.events`(원본)와 **섞지 않습니다** — 원본은 수정하지 않고, 학생 메모는 분석 문서에만 씁니다. 그래프에서 원본 사건선과 학생 메모선을 색으로 구분합니다(원본=회색, 학생 메모=빨강).
- **`visibleExps`가 없으면 `[1, 2, 3]`으로 봅니다** (v2.0). 기존 학급 문서에는 이 필드가 없으므로, 없을 때 기본값을 적용하지 않으면 탭이 전부 사라집니다. 학생 화면(`index.html`)과 교사 화면(`teacher.js`) **양쪽 모두** 이 기본값 처리를 해야 합니다. 실험 4(비열)는 기본값에 없으므로 **교사가 켜야 학생에게 보입니다.**
- **학급 생성 순서 (v2.1)** — 보안 규칙상 `joinCodes` 문서를 만들려면 그 `classId`의 학급 문서에서 `teacherUid`를 확인합니다. 따라서 **`classes` 문서를 먼저 만들고, 그다음 `joinCodes` 문서를 만들어야** 합니다. 두 쓰기를 하나의 배치(batch)로 묶으면 학급 문서가 아직 안 보여 규칙 검사에 실패할 수 있으므로 **순차로 씁니다.**
- `classes/{classId}/groups/{groupId}`는 **현재 어느 코드도 읽지 않습니다** (모둠 아이디는 `auth.js`가 `"g" + 모둠번호`로 만들고, 화면 표시는 문자열 변환으로 처리). 학급을 만들 때 이 하위 컬렉션을 만들 필요가 없습니다.
- **학급을 새로 만들 때는 `classes` 문서와 `joinCodes/{코드}` 문서를 함께 만듭니다** (v1.7). 둘 중 하나만 있으면 학생이 입장할 수 없습니다. 입장 코드를 바꿀 때는 옛 `joinCodes` 문서를 지우고 새 코드로 다시 만든 뒤, `classes.joinCode` 필드도 같이 고칩니다.

### 5.3 인증

| 대상 | 방식 |
|---|---|
| 학생 | Firebase **익명 인증** + 학급 코드(`joinCode`) + 모둠 선택 |
| 교사 | Firebase **Google 로그인** |

학생은 계정을 만들지 않습니다. 최초 1회 학급 코드와 모둠을 입력하면 `localStorage`에 저장되어 이후 자동 입장합니다.

**입장 절차 (v1.7)** — 학생이 코드를 넣으면 `joinCodes/{입력한 코드}` 문서 **하나만** 읽어 `classId`를 얻고, 그 학급 문서를 읽습니다. `classes` 컬렉션 전체를 검색(`list`)하지 않습니다. 코드가 틀리면 문서가 없으므로 "코드를 찾을 수 없다"로 처리합니다.

### 5.4 보안 규칙 요지

- `joinCodes` 읽기: 인증된 사용자가 **문서 하나 지정(`get`)해서만** 가능. 목록 조회(`list`)는 차단
- `classes` 읽기 (v1.8):
  - `get`: 인증된 사용자면 가능 (학생이 자기 학급 이름을 불러올 때)
  - `list`: **조건부 허용** — `resource.data.teacherUid == request.auth.uid`일 때만. 즉 교사가 `where("teacherUid","==",본인)`으로 거는 쿼리는 통과하고, 학생이 조건 없이 전체를 긁는 쿼리는 거부됩니다
- `datasets` 읽기: 인증된 사용자면 가능(학급 비교 기능에 필요)
- `datasets` 쓰기: 생성은 인증된 사용자(`ownerUid`가 본인이어야 함), 수정·삭제는 `ownerUid` 본인 또는 교사
- `classes`·`joinCodes` 문서 생성·수정: 교사만
- 인증되지 않은 접근: 전면 차단
- 쓰기 검증: `points`는 5,000개 이하, `expNo`는 1~3만 허용 (§5.2 한도를 규칙으로도 막습니다)

> **왜 `list`를 조건부로 여는가 (v1.8)** — Firestore 규칙은 목록 쿼리를 "결과로 나올 수 있는 모든 문서가 규칙을 만족하는가"로 판정합니다. 따라서 `allow list: if resource.data.teacherUid == request.auth.uid`라고 쓰면, 교사의 `where("teacherUid","==",본인)` 쿼리는 결과가 전부 조건을 만족하므로 **통과**하고, 조건 없는 전체 조회는 남의 학급이 섞일 수 있으므로 **거부**됩니다. v1.7에서 `list`를 전면 차단으로 적었으나, 그러면 교사 대시보드(`js/teacher.js`)가 자기 학급 목록을 불러오지 못해 바로잡았습니다.

### 5.5 익명 인증의 한계 (v1.7)

**규칙만으로는 막을 수 없는 것이 있습니다.** 학생은 계정 없이 익명 인증으로 들어오므로, Firestore 규칙 입장에서는 "이 사람이 몇 반 몇 모둠인지" 증명할 방법이 없습니다. 따라서 아래는 규칙으로 해결하지 않고 다른 방법으로 보완합니다.

| 남는 위험 | 왜 규칙으로 못 막나 | 보완 방법 |
|---|---|---|
| 같은 학급 안에서 다른 모둠의 `analyses`를 덮어씀 | 익명 uid로는 모둠 소속을 증명할 수 없음. `ownerUid`로 잠그면 브라우저 데이터가 지워졌을 때 **자기 모둠이 자기 분석을 못 고치게 되는** 더 큰 사고가 남 | 화면에서 다른 모둠 분석을 열 수 없게 하고, 교사가 대시보드에서 주기적으로 확인 |
| 코드를 아는 사람이 그 학급 데이터를 읽음 | 코드 = 유일한 신원 증명이므로 | 학급 코드를 외부에 공개하지 않음. 코드가 새면 `joinCodes` 문서를 교체 |

이 한계를 감수할 수 있는 이유는 **절대 규칙 3·4** 때문입니다. 실명을 저장하지 않고 건강정보를 개인 식별자와 결합하지 않으므로, 최악의 경우 새어나가는 것은 "4모둠의 심박수 132bpm"처럼 **누구인지 알 수 없는 값**입니다.

### 7.1 그래프 좌표 확인 기능

**모든 그래프는 마우스를 데이터 위에 올리면 정확한 좌표값이 보여야 합니다.** 3단계(패턴 찾기)에서 "몇 분에 몇 ppm이었나"를 정확히 짚으려면 눈짐작이 아니라 실제 수치 확인이 필요하기 때문입니다.

`renderChart(canvasEl, spec)`는 Chart.js의 기본 tooltip 기능을 켜서 구현합니다. 별도 라이브러리는 추가하지 않습니다.

**동작 요구사항**

- 그래프 위 아무 지점에 마우스를 올리면 가장 가까운 데이터 점이 강조되고, 그 점의 `(시간, 값)`이 말풍선으로 뜬다
- 시간은 `경과 초`를 사람이 읽는 `분:초` 형태로 변환해서 보여준다 (예: `125초` → `2분 05초`)
- 값에는 단위를 붙인다 (예: `842 ppm`)
- 이벤트 마커(`events`, "창문 열기" 등) 근처에 마우스를 올리면 이벤트 라벨도 함께 보여준다
- 산점도(실험 3)에서는 두 축의 값을 각각 보여준다 (예: `운동 강도: 3단계 · 심박수: 132 bpm`)
- 터치 기기(모둠 노트북이 터치스크린인 경우)에서는 탭으로도 같은 정보가 뜨게 한다
- 모둠 노트북 화면에서 잘 보이도록 툴팁 글자 크기는 최소 13px 이상으로 한다

**`spec.tooltip` 옵션 (config 파일에서 실험별로 지정)**

```javascript
tooltip: {
  timeFormat: "mmss",        // "mmss" | "sec" | "clock"
  valueLabel: "CO2 농도",     // 값 앞에 붙는 이름
  valueUnit: "ppm",
  extra: (point) => ""        // 실험별 추가 정보 (예: 실험3의 조건 라벨)
}
```

이 옵션은 §10의 각 실험 config에 이미 반영돼 있습니다.

## 8. 분석 단계 — 모든 실험 탭의 공통 뼈대

**이 절은 담당과 무관하게 모든 세션이 읽습니다.**

### 8.1 왜 단계로 나누는가

학생에게 그래프만 던져주면 "선이 올라갔다"에서 멈춥니다. 결론에 이르는 사고 과정을 단계로 쪼개고 각 단계에서 질문을 던져야 "왜 올라갔고, 그래서 무엇을 해야 하는가"까지 갑니다.

이 5단계는 2022 개정 과학과 교육과정의 **과정·기능** 범주를 따릅니다(§9).

### 8.2 5단계

| 단계 | 이름 | 학생이 하는 일 | 대응 과정·기능 |
|---|---|---|---|
| **1** | 데이터 살펴보기 | 무엇을 어떻게 쟀는지 확인, 이상한 값 점검 | 자료 수집 |
| **2** | 그래프로 나타내기 | 그래프 종류 선택, 축·기준선 설정 | 자료 변환 |
| **3** | 패턴 찾기 | 그래프에서 읽어낸 사실 기록 (좌표 확인 기능 활용) | 자료 분석·해석 |
| **4** | 결론 내리기 | 처음 질문에 대한 답 작성 + 근거 제시 | 결론 도출 |
| **5** | 되돌아보기 | 한계, 다음 질문, 일반화 가능성 | 일반화·의사소통 |

### 8.3 각 단계의 화면 구성

```
┌─ 3단계  패턴 찾기 ─────────────────────────┐
│                                            │
│  [그래프 영역]                              │
│   ※ 그래프 위에 마우스를 올리면              │
│      정확한 시간·값이 말풍선으로 뜬다         │
│                                            │
│  💡 이런 걸 살펴보세요                       │
│   · 값이 가장 빠르게 변한 구간은 어디인가요?   │
│   · 기준선을 넘은 건 몇 분쯤인가요?           │
│     (그래프에 마우스를 올려 정확히 확인해 보세요)│
│                                            │
│  ┌────────────────────────────────────┐   │
│  │ 우리가 발견한 것을 써 보세요...       │   │
│  └────────────────────────────────────┘   │
│                                            │
│  [자동 계산]  최댓값 1,180 ppm             │
│               상승 속도 18 ppm/분           │
│                                            │

└────────────────────────────────────────────┘
```

### 8.4 단계 UI 규칙

- 단계는 **순서대로 진행**하되, 이전 단계로 돌아갈 수 있습니다.
- 학생 답변은 입력할 때마다 자동 저장합니다(디바운스 1초). "저장" 버튼을 누르게 하지 않습니다.
- **비계 질문은 답을 알려주지 않습니다.** "1,000ppm을 넘었네요"가 아니라 "기준선을 넘은 건 몇 분쯤인가요?"라고 묻습니다.
- 자동 계산값(최댓값, 평균, 기울기 등)은 **보여주되 해석하지 않습니다.** 해석은 학생 몫입니다.
- 답변 칸이 비어 있어도 다음 단계로 갈 수 있습니다. 강제하지 않습니다.
- 4단계 결론에는 **"근거가 된 그래프 부분"을 지목**하게 합니다. 근거 없는 주장을 막는 장치입니다. §7.1의 좌표 확인 기능으로 짚은 지점을 그대로 인용하도록 유도합니다 (예: "2분 05초, 1,020ppm 지점에서 기준선을 넘었다").

### 8.5 AI 활용 기록

각 단계에 "AI에게 물어보기" 영역을 둡니다. 단, **AI 답변을 그대로 넣지 못하게** 합니다.

기록 항목:
- 무엇을 물었나
- AI가 뭐라고 했나 (요약)
- **우리 데이터와 맞는가?** (맞음 / 다름 / 판단 불가) ← 필수 선택
- 다르다면 왜 그렇게 판단했나

이 기록이 `analyses.aiLog`에 저장되고, 평가 근거이자 포스터의 필수 항목이 됩니다.

### 10.1 실험 1 — 우리 교실 최적 환기 주기 찾기 [세션 B]

**탐구 질문:** 우리 교실은 몇 분마다 환기해야 할까?

| 항목 | 값 |
|---|---|
| 센서 | CO₂(ppm), 온도(℃) |
| 데이터 형태 | 시계열 — 시간에 따른 CO₂ 변화 |
| 기본 그래프 | 선그래프 |
| 기준선 | **1,000 ppm** (학교보건법 시행규칙 별표 4의2, 교사 내 이산화탄소 유지기준) |
| 조건 라벨 | 창문 닫음 / 창문 열음 / 쉬는 시간 / 문만 열음 |
| 좌표 툴팁 | 시간(분:초) + CO₂(ppm), 이벤트 마커 근처는 이벤트명도 표시 |
| 특수 기능 | **이벤트 마커** — "창문 열기" 등을 그래프 위 수직선으로 표시 |

**자동 계산 항목**
- 최댓값, 최솟값
- 상승 속도 (ppm/분) — 창문 닫힌 구간의 기울기
- 기준선 도달 시각 (몇 분 만에 1,000ppm을 넘었나)
- 환기 후 회복 시간 (창문 연 뒤 기준선 아래로 내려가기까지)

**비계 질문**

| 단계 | 질문 |
|---|---|
| 1 데이터 살펴보기 | · 몇 분 동안 측정했나요? 몇 초 간격으로 쟀나요?<br>· 측정하는 동안 교실에서 어떤 일이 있었나요?(사람 이동, 문 열림 등)<br>· 값이 갑자기 튄 구간이 있나요? 있다면 그때 무슨 일이 있었을까요? |
| 2 그래프로 나타내기 | · 우리 데이터는 '시간에 따라 변하는 값'입니다. 어떤 그래프가 어울릴까요?<br>· 기준선 1,000ppm을 그래프에 그어 봅시다. 왜 이 선이 중요할까요? |
| 3 패턴 찾기 | · 그래프에 마우스를 올려가며, CO₂가 가장 빠르게 올라간 구간을 정확한 시각으로 찾아보세요.<br>· 기준선을 넘은 것은 정확히 몇 분 몇 초인가요?<br>· 창문을 연 뒤 몇 분 만에 내려갔나요?<br>· 올라갈 때와 내려갈 때, 어느 쪽이 더 빠른가요? 왜 그럴까요? |
| 4 결론 내리기 | · **우리 교실은 몇 분마다 환기해야 할까요?**<br>· 그렇게 판단한 근거는 그래프의 어느 지점(시각·값)인가요?<br>· 환기는 몇 분 동안 해야 충분할까요? |
| 5 되돌아보기 | · 다른 반에서도 같은 결과가 나올까요? 무엇이 다르면 결과가 달라질까요?<br>· 우리 측정에서 아쉬웠던 점은 무엇인가요?<br>· 교실의 CO₂는 대체 어디서 온 걸까요?(→ 실험 3에서 다시 만납니다) |

**산출물:** 교실 환기 가이드 — 그래프 + 권장 환기 주기가 담긴 게시물 이미지

### 10.2 실험 2 — 교실 식물 최적 조건 찾기 [세션 C]

**탐구 질문:** 식물이 가장 잘 광합성하는 조건은?

| 항목 | 값 |
|---|---|
| 센서 | CO₂(ppm), 조도(lux), 온도(℃) |
| 데이터 형태 | 조건별 비교 — 여러 조건에서 각각 시계열 측정 |
| 기본 그래프 | 막대그래프(조건별 변화량), 보조로 선그래프(원본 시계열) |
| 기준선 | 없음. 대신 **대조군(빛 없음) 기준선** 표시 |
| 조건 라벨 | 학생이 직접 정의 (예: 어두움 / 보통 / 밝음, 또는 빛 색깔별) |
| 좌표 툴팁 | 막대: 조건명 + 변화량(ppm). 선그래프: 시간(분:초) + CO₂(ppm) |
| 특수 기능 | **변인 기록** — 조작 변인, 통제 변인 입력란 |

**자동 계산 항목**
- 조건별 CO₂ 변화량 (시작값 − 끝값)
- 조건별 변화 속도 (ppm/분)
- 조건 간 차이
- 반복 측정이 있으면 평균과 편차

**비계 질문**

| 단계 | 질문 |
|---|---|
| 1 데이터 살펴보기 | · 무엇을 다르게 했나요?(조작 변인)<br>· 무엇을 똑같이 유지했나요?(통제 변인) 빠뜨린 것은 없나요?<br>· 각 조건을 몇 번씩 반복했나요? 한 번만 쟀다면 그 결과를 믿을 수 있을까요? |
| 2 그래프로 나타내기 | · 조건끼리 비교하려면 어떤 그래프가 좋을까요?<br>· 세로축에 무엇을 놓아야 비교가 잘 될까요?(농도 자체? 변화량?) |
| 3 패턴 찾기 | · 막대에 마우스를 올려 정확한 변화량을 확인해 보세요. 어느 조건에서 CO₂가 가장 많이 줄었나요?<br>· 조건 사이의 차이가 뚜렷한가요, 비슷한가요?<br>· 이 차이가 우연일 가능성은 없을까요? 어떻게 확인할 수 있을까요?<br>· CO₂가 오히려 늘어난 조건이 있나요? 있다면 왜일까요? |
| 4 결론 내리기 | · **빛의 세기와 광합성은 어떤 관계인가요?**<br>· 근거가 된 데이터(어느 조건의 몇 ppm 변화)는 무엇인가요?<br>· 빛을 계속 밝게 하면 광합성도 계속 늘어날까요? 우리 데이터로 답할 수 있나요? |
| 5 되돌아보기 | · 어두운 조건에서 CO₂가 늘었다면, 식물이 무엇을 하고 있는 걸까요?<br>· 교실에 식물을 두면 공기가 좋아질까요? 실험 1의 데이터와 비교해 봅시다.<br>· 우리 실험에서 통제하지 못한 변인은 무엇인가요? |

**산출물:** 조건별 비교 그래프 + 교실 식물 배치 제안

### 10.3 실험 3 — 운동과 우리 몸 [세션 D]

**대주제:** 운동할 때 우리 몸은 어떻게 반응할까?

| 항목 | 값 |
|---|---|
| 센서 | 심박수(bpm), 폐활량(mL), CO₂(ppm) |
| 데이터 형태 | 개인별 반복 측정 + 학급 분포 |
| 기본 그래프 | 산점도(두 지표 관계), 선그래프(회복 곡선) |
| 기준선 | 없음. 대신 **학급 평균선** |
| 조건 라벨 | 안정 시 / 가벼운 운동 후 / 심한 운동 후 / 회복 1분·2분·3분·5분 |
| 좌표 툴팁 | 산점도: 두 축의 값을 각각 표시 (예: "운동 강도: 3단계 · 심박수: 132 bpm"). 선그래프: 경과 시간 + 값 |
| 특수 기능 | **학급 분포에서 내 모둠 위치 표시** (익명) |

**모둠별 선택 주제** — 모둠마다 다른 질문을 고를 수 있게 합니다.
1. 운동 강도와 심박수의 관계
2. 심박수 회복 속도의 개인차
3. 폐활량과 신체 조건의 관계
4. 운동 전후 날숨 CO₂ 변화
5. 호흡수와 호흡량의 변화
6. 심박수와 호흡수의 상관관계

**자동 계산 항목**
- 개인별 평균, 최댓값
- 회복 시간 (안정 시 수준으로 돌아오기까지)
- 두 변수 간 상관 경향 (수치가 아니라 "함께 증가하는 경향" 수준의 표현)
- 학급 분포에서의 백분위 (익명)

**비계 질문**

| 단계 | 질문 |
|---|---|
| 1 데이터 살펴보기 | · 몇 명의 데이터인가요? 각자 몇 번씩 측정했나요?<br>· 측정 조건이 모두 같았나요?(운동 시간, 측정 시점)<br>· 유난히 다른 값이 있나요? 측정 실수일까요, 진짜 개인차일까요? |
| 2 그래프로 나타내기 | · 두 값의 '관계'를 보려면 어떤 그래프가 좋을까요?<br>· 시간에 따른 '변화'를 보려면요?<br>· 우리 모둠 데이터와 학급 전체를 함께 보려면 어떻게 해야 할까요? |
| 3 패턴 찾기 | · 점 하나하나에 마우스를 올려 정확한 값을 확인해 보세요. 운동 강도가 올라갈 때 심박수는 어떻게 변했나요?<br>· 사람마다 반응이 얼마나 달랐나요?<br>· 두 지표가 함께 움직이나요? 아니면 따로 노나요?<br>· 회복이 빠른 사람과 느린 사람의 차이는 무엇일까요? |
| 4 결론 내리기 | · **운동할 때 우리 몸에서 무슨 일이 일어나나요?** 소화·순환·호흡을 연결해 설명해 봅시다.<br>· 심박수가 올라가는 이유는 무엇인가요? 세포호흡과 연결해 보세요.<br>· 근거가 된 데이터(정확한 수치)는 무엇인가요? |
| 5 되돌아보기 | · **3월에 잰 교실 CO₂를 기억하나요? 그 CO₂는 어디서 왔을까요?**<br>· 우리 데이터로 말할 수 있는 것과, 말할 수 없는 것은 무엇인가요?<br>· 더 알아보고 싶은 것은 무엇인가요? |

**산출물:** AI 연구 포스터용 그래프 + 결론 + AI 활용 기록

**주의:** 이 실험은 건강정보를 다룹니다. 실명 저장 금지, 학급 비교는 익명 라벨만, 개인 데이터는 본인 모둠만 조회 가능하게 합니다.

### 10.4 실험 4 — 물과 식용유, 어느 쪽이 빨리 뜨거워질까 (비열) [세션 B, v2.0]

**추가 실험입니다.** 실험 1·2·3과 달리 17차시 계획에 들어 있지 않고, **교사가 켜야 학생 화면에 보입니다**(`classes.visibleExps`). 온도 센서 2대만 있으면 되므로, CO₂ 센서 입고 전에도 실물 데이터로 수업·검증이 가능한 것이 도입 이유입니다.

**탐구 질문:** 같은 열을 주면 물과 식용유 중 어느 쪽이 더 빨리 뜨거워질까?

| 항목 | 값 |
|---|---|
| 센서 | 온도(℃) **2대** — 물질마다 1대씩 |
| 데이터 형태 | 시계열 — 시간에 따른 온도 상승 (실험 1과 같은 구조) |
| 기본 그래프 | 선그래프 (두 물질을 겹쳐 그림) |
| 기준선 | 없음 |
| 조건 라벨 | 물질 이름 (물 / 식용유 / …) |
| 좌표 툴팁 | 시간(분:초) + 온도(℃) |
| 특수 기능 | **두 곡선의 기울기(온도 상승 속도) 비교**. 측정은 수집기에서 센서 2대 동시 진행(§11 세션 F) |

**자동 계산 항목**
- 시작 온도, 끝 온도, 온도 변화량(ΔT)
- 온도 상승 속도 (℃/분) — 두 물질의 핵심 비교값
- 같은 시각에서 두 물질의 온도 차이

**비계 질문 (원본 — §12.2에 따라 담당 세션이 다시 설계할 것)**

| 단계 | 질문 |
|---|---|
| 1 데이터 살펴보기 | · 두 물질을 각각 몇 분 동안 쟀나요? 같은 시간 동안 쟀나요?<br>· 시작 온도가 서로 비슷했나요? 달랐다면 비교할 때 무엇을 조심해야 할까요?<br>· 두 컵에 담은 양은 같았나요? 불의 세기는 같았나요? |
| 2 그래프로 나타내기 | · 두 물질의 온도 변화를 한 그래프에 겹쳐 그려 보세요. 어느 쪽 선이 더 가파른가요?<br>· 시작 온도가 다르다면, 온도 자체와 온도 변화량 중 무엇으로 비교해야 공평할까요? |

## 12. 각 세션이 스스로 고민해서 반영할 것

**이 절은 지시사항입니다.** §10의 비계 질문은 **출발점이지 완성본이 아닙니다.** 각 세션은 담당 실험을 맡은 뒤 아래를 직접 고민하고 결과를 코드와 config에 반영하십시오.

### 12.1 데이터를 먼저 보라

명세만 읽고 화면을 만들지 마십시오. `generateMock()`으로 **담당 실험의 가짜 데이터를 실제로 생성해서 그래프로 그려 보고**, 그 그래프를 보면서 다음을 판단하십시오.

- 이 그래프에서 중학교 2학년이 **실제로 읽어낼 수 있는 것**은 무엇인가?
- 반대로, 읽어내기 어려운 것은 무엇인가? 그것을 도우려면 어떤 보조 표시(기준선, 구간 음영, 주석, **좌표 툴팁**)가 필요한가?
- 자동 계산값 중 학생의 결론에 실제로 쓰이는 것은 무엇인가? 쓰이지 않는 숫자는 화면에서 빼라.

### 12.2 비계 질문을 다시 설계하라

§10의 질문을 그대로 복사하지 말고, 아래 기준으로 검토·수정·추가하십시오.

1. **답을 알려주는 질문이 아닌가?** "1,000ppm을 넘었지요?"는 실패한 질문입니다. "기준선을 넘은 건 몇 분쯤인가요?"가 맞습니다.
2. **데이터를 보지 않고도 답할 수 있는 질문이 아닌가?** 그렇다면 삭제하십시오. 모든 질문은 학생이 자기 그래프를 들여다봐야 답할 수 있어야 합니다.
3. **단계마다 사고 수준이 올라가는가?** 1~2단계는 확인, 3단계는 관찰, 4단계는 추론과 판단, 5단계는 성찰과 전이입니다. 4단계에 관찰 수준 질문이 있으면 잘못 배치된 것입니다.
4. **막힐 학생을 위한 추가 힌트가 필요한가?** 필요하다면 "더 힌트 보기" 형태로 접어 두십시오. 처음부터 다 보이면 스스로 생각하지 않습니다.
5. **질문 수가 적절한가?** 한 단계에 5개를 넘으면 학생이 지칩니다. 3~4개로 줄이고 가장 중요한 것을 앞에 두십시오.

### 12.3 결론에 이르는 길을 확인하라

가장 중요한 점검입니다. **직접 학생 입장이 되어 1단계부터 5단계까지 따라가 보십시오.** 그리고 답하십시오.

- 이 단계들을 순서대로 밟으면 **탐구 질문에 대한 답이 실제로 나오는가?**
- 중간에 논리적 도약이 있지 않은가? 있다면 그 사이에 단계나 질문을 넣으십시오.
- 학생이 4단계에서 "그래서 결론이 뭐지?"라고 막힌다면, 3단계에서 필요한 관찰을 충분히 끌어내지 못한 것입니다. 3단계로 돌아가 고치십시오.

결론이 나오지 않는 구조라면 그것은 명세의 결함입니다. **사용자에게 알리고 수정을 제안하십시오.** 명세를 그대로 구현하는 것보다 중요합니다.

### 12.4 과학과 과정·기능을 반영하라

§9의 5개 범주 중 **담당 실험에서 특히 강조할 것 2개**를 고르고, 그것이 화면에서 어떻게 드러나는지 config 주석에 적으십시오. 예: "이 실험은 '탐구 설계 및 수행'을 강조한다 → 변인 기록 UI를 1단계 앞에 배치했다."

전부 담을 필요는 없습니다. 담당 실험의 성격에 맞는 것을 골라 확실히 반영하는 편이 낫습니다.

### 12.5 중학생의 언어로 쓰라

모든 화면 문구를 소리 내어 읽어 보십시오. 중학교 2학년이 이해하지 못할 단어가 있으면 바꾸십시오.

- "변인을 통제하시오" → "다른 조건은 똑같이 맞췄나요?"
- "상관계수" → "두 값이 함께 움직이는 정도"
- "이상치" → "혼자 튀는 값"

### 12.6 좌표 확인 기능을 탐구에 실제로 연결하라

§7.1의 툴팁은 기능으로 넣는 것으로 끝이 아닙니다. §10의 비계 질문 중 "정확한 시각", "정확한 값"을 묻는 질문들이 **이 기능을 실제로 쓰게 만드는지** 확인하십시오. 질문이 "대략 언제쯤"에 머물러 있으면 학생은 툴팁을 켤 이유가 없습니다. 정확한 수치를 답에 쓰도록 요구해야 기능이 살아납니다.

