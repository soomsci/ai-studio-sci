# 과학 데이터 스튜디오

중학교 2학년 과학 프로젝트 수업용 데이터 수집·분석 웹앱.
학생이 PASCO 센서로 모은 데이터를 스스로 분석해 결론을 도출하도록 안내하는 것이 목적이다.

## 시작하기 전에

**작업 시작 전 `docs/SPEC.md`를 읽는다. 단, 전체를 다 읽지 않는다.**

- §0, §3(절대 규칙), §5(데이터 계약)는 모든 세션이 읽는다.
- 그 외는 `docs/SPEC.md` §11.0의 "필독 절" 표에 적힌, **자기 세션 담당 절만** 읽는다.
- 다른 실험(§10.1/10.2/10.3 중 자기 담당이 아닌 것)은 읽지 않는다. 필요해지면 그때 읽는다.

특히 아래 세 곳은 담당 여부와 관계없이 반드시 읽는다.

- §5 데이터 계약 — 세션 간 약속. 임의 변경 금지
- §8 분석 단계 구조 — 모든 실험 탭의 공통 뼈대
- §12 각 세션이 고민할 것 — 명세를 그대로 옮기지 말라는 지시

## 파일 소유권

세션마다 담당 파일이 정해져 있다. `docs/SPEC.md` §4를 확인하고 **자기 담당 파일만 수정한다.**

| 세션 | 담당 |
|---|---|
| A | 공통 뼈대, 홈, `js/` 공통 모듈, `mock/` |
| B | `js/exp1.js`, `config/exp1.config.js` |
| C | `js/exp2.js`, `config/exp2.config.js` |
| D | `js/exp3.js`, `config/exp3.config.js` |
| E | `teacher.html`, `js/teacher.js` |
| F | `collector/` |
| **G** | **`docs/PROGRESS.md`, git 커밋/브랜치 관리. 다른 파일은 읽기만 한다** |

담당 밖의 파일을 고쳐야 할 상황이면 **작업을 멈추고 사용자에게 알린다.** 직접 고치지 않는다. **세션 G는 예외 없이 이 규칙을 지킨다** — 문제를 발견해도 다른 세션의 파일을 직접 고치지 않고, 무엇을 고쳐야 하는지 정리해서 보고한다.

## 기술 규칙

- **순수 HTML + CSS + JavaScript.** 프레임워크·번들러·TypeScript 도입 금지
- `npm install` 하지 않는다. 외부 라이브러리는 Chart.js(CDN)만 사용
- Firebase: Firestore, Auth(익명 + Google), Hosting
- 파이썬은 3.11 (`pasco` 라이브러리 요구사항)

유지보수 주체가 교사 1인이다. 파일을 열어서 바로 고칠 수 있는 상태를 유지한다.

## 절대 규칙

1. `docs/SPEC.md` §5 스키마를 변경하지 않는다
2. **학생 실명을 저장하지 않는다.** 모둠 단위 익명 라벨만 사용
3. 건강정보(심박수·폐활량)를 개인 식별자와 결합하지 않는다
4. 측정 데이터는 Firestore에. `localStorage`는 학급 코드·모둠 번호·화면 설정만
5. 측정 1회 = `datasets` 문서 1개. 데이터 포인트마다 문서를 만들지 않는다
6. 원본 측정값은 수정하지 않는다. 분석은 별도 문서에 저장한다
7. **Firebase API 키 등 설정값을 공개 GitHub 저장소에 커밋하지 않는다.** `.gitignore`에 설정 파일을 포함한다

## 화면 문구

- 모든 UI 텍스트는 한국어. **중학교 2학년이 읽는 문장**으로 쓴다
- 화면에 영어를 노출하지 않는다 (용어 대응표는 SPEC §14)
- 전문 용어는 풀어 쓴다: "이상치" → "혼자 튀는 값"
- 비계 질문은 **답을 알려주지 않는다.** 학생이 자기 데이터를 봐야 답할 수 있는 질문만 쓴다

## 센서가 아직 없다

실제 센서 없이 개발한다. `mock/mock-data.js`의 `generateMock()`으로 가짜 데이터를 만들어 쓴다.
가짜 데이터와 실제 데이터의 형태가 같으므로, 나중에 수집기만 붙이면 된다.

**화면을 만들기 전에 반드시 가짜 데이터를 실제로 그려 보고 시작한다.** (SPEC §12.1)

## 총괄 세션(G) 사용법

세션 G는 A~F와 성격이 다르다. 기능을 만들지 않고 **전체를 점검하고 진행 상황을 정리**한다.

### G가 할 수 있는 일 / 할 수 없는 일

- G는 A~F **각 세션의 대화 내용을 볼 수 없다.** 같은 폴더를 공유할 뿐, 세션끼리는 서로의 대화를 모른다.
- G가 볼 수 있는 것은 **디스크에 남은 결과물**뿐이다: 파일 내용, git 커밋 이력, 실행 결과.
- 그래서 A~F는 **작업 한 덩어리가 끝날 때마다 커밋**해야 한다. 커밋이 없으면 G는 진행 상황을 알 방법이 없다.

### G의 정기 점검 항목

1. `git log`, `git status`로 최근 변경 사항 확인
2. 각 세션이 **자기 담당 파일만** 건드렸는지 확인 (`docs/SPEC.md` §4 기준)
3. `docs/SPEC.md` §3(절대 규칙), §5(데이터 계약)를 어긴 곳이 없는지 확인 — 특히 스키마 임의 변경, 실명 저장, `datasets` 문서 분할 여부
4. 통합 지점 점검 — `index.html`에 각 실험 탭 스크립트가 올바르게 연결됐는지, 세션 A가 만든 함수 시그니처를 B·C·D가 실제로 맞게 호출하는지
5. 가능하면 로컬에서 페이지를 열어 콘솔 에러가 없는지 확인
6. 위에서 발견한 문제를 **직접 고치지 않고** `docs/PROGRESS.md`에 정리하고 사용자에게 보고
7. 문제가 없는 구간은 커밋을 정리(필요하면 사용자 승인 하에 병합)하고 `docs/PROGRESS.md`를 갱신

### `docs/PROGRESS.md` 형식

```markdown
## 2026-07-20 점검
- [x] 세션 A: data.js, chart-kit.js, steps.js 완료. §7 시그니처 확인함
- [ ] 세션 B: exp1.config.js까지 완료, exp1.js 화면 골격 진행 중
- [ ] 세션 C, D: 세션 B 완료 대기 중 (§11.0 권장 순서)
- 문제 발견: index.html에 exp1.js <script> 태그 누락 → 세션 A 또는 B에 전달 필요
```

이 파일 하나만 보면 사용자가 전체 진행 상황을 파악할 수 있어야 한다. A~F 각 세션에도 "작업 한 덩어리가 끝나면 커밋하고, 무엇을 했는지 한 줄로 커밋 메시지에 남겨라"라고 안내되어 있다.

### 변경 요청 라우팅

사용자가 웹앱에서 뭔가 바꾸고 싶을 때, **어느 파일을 어느 세션이 담당하는지 사용자가 직접 판단할 필요가 없다.** 사용자는 G에게 자연어로 요청만 하고, G가 판단해서 붙여넣을 프롬프트를 써준다.

**처리 절차**

1. 요청 내용을 읽고, 그것이 어떤 화면·기능에 해당하는지 파악한다.
2. `docs/SPEC.md` §4(폴더 구조와 파일 소유권) 표에서 해당 파일과 담당 세션을 찾는다.
3. 이 변경이 `docs/SPEC.md` §5(데이터 계약)나 §3(절대 규칙)에 영향을 주는지 확인한다.
   - **영향을 준다면 먼저 사용자에게 알린다.** 예: "이 요청은 Firestore 스키마를 바꿔야 해서, 실행하면 이미 저장된 데이터와 안 맞을 수 있습니다. SPEC.md를 먼저 고칠까요?" 스키마·절대 규칙 변경은 모든 세션에 영향을 주므로 조용히 진행하지 않는다.
   - 영향이 없다면 4번으로 진행한다.
4. 담당 세션에 그대로 붙여넣을 수 있는 **완성된 프롬프트**를 작성한다. 아래 형식을 따른다.
5. 변경이 여러 세션의 파일에 걸치면(예: 공통 함수 추가는 세션 A, 그 함수를 쓰는 화면 변경은 세션 B), **세션별로 프롬프트를 나누어 작성**하고 어떤 순서로 진행해야 하는지 명시한다(의존 관계가 있는 쪽을 먼저).
6. 담당 세션이 불분명하면(예: 완전히 새로운 화면) 추측해서 단정하지 말고 사용자에게 확인한다.
7. 라우팅 결과 `docs/SPEC.md`나 `CLAUDE.md` 자체를 고쳐야 한다면(예: 데이터 스키마 변경, 새 공통 규칙), **G가 직접 그 문서를 수정한다.** 버전 번호를 올리고 부록 B(SPEC.md) 변경 이력에 한 줄을 남긴다. 사용자에게 매번 문서 수정을 요청할 필요 없이, G가 실제 저장소에 접근할 수 있으므로 이 역할을 맡는다.

**프롬프트 작성 형식**

```
━━━ 세션 [B] 창에 붙여넣기 ━━━

CLAUDE.md의 "작업 방식 — 토큰·비용 절약"을 지켜라.
[변경 내용을 구체적으로: 무엇을, 어느 화면에서, 어떻게]

관련 절: docs/SPEC.md §[해당 절만 지정]
(전체를 다시 읽지 말고 이 범위만 확인해라)

담당 파일: [해당 세션이 원래 담당하는 파일]만 수정해라.
[필요하면 제약 조건이나 주의사항 추가]
```

이 형식을 벗어나 애매하게 요약만 던지지 않는다. **받는 세션이 다른 설명 없이 그대로 실행할 수 있을 정도로 구체적으로** 쓴다.

**예시**

사용자: "실험 1 그래프에 평균선도 하나 더 그어줬으면 좋겠어"

G의 판단: 그래프 렌더링은 `chart-kit.js`(세션 A 소유)의 `renderChart` 함수 기능이지만, "평균선을 켤지"는 실험 1의 설정이므로 `exp1.config.js`(세션 B 소유)에서 옵션을 넘기는 방식이 맞다. `chart-kit.js`에 평균선 지원 자체가 없다면 세션 A 작업이 먼저 필요하다.

→ `chart-kit.js`에 평균선(`avgLine`) 옵션이 이미 있는지 먼저 확인이 필요하다고 사용자에게 안내하거나, 없다면 세션 A용 프롬프트(옵션 추가)와 세션 B용 프롬프트(옵션 사용)를 순서대로 작성해 제시한다.

### §12 이행 감사

`docs/SPEC.md` §12는 각 세션에게 §10의 비계 질문을 그대로 베끼지 말고 직접 재설계하라고 지시한다. 이게 실제로 지켜졌는지는 아무도 강제로 확인하지 않으면 넘어가기 쉽다. G가 이 부분을 점검한다.

- `config/exp1.config.js`(및 exp2·exp3)의 `steps` 질문 문구를 `docs/SPEC.md` §10.1(또는 10.2·10.3)의 원본 비계 질문과 **비교**한다. 문장이 원본과 거의 동일하면 §12.2 재설계가 이루어지지 않았다는 신호이므로 `docs/PROGRESS.md`에 기록하고 해당 세션에 재검토를 요청하는 프롬프트를 작성한다.
- `mock/mock-data.js`의 `generateMock()`으로 담당 실험 데이터를 만들어, 1단계부터 5단계까지 **직접 따라가며 결론이 실제로 나오는지** 확인한다(§12.3과 동일한 검증을 세션이 했는지 재확인하는 것). 막히는 지점이 있으면 기록한다.
- config 파일에 §12.4(과학과 과정·기능 반영)에 대한 주석이 있는지 확인한다. 없으면 있으면 좋다는 정도로만 기록하고, 필수 차단 사유로는 삼지 않는다.

### 미해결 위험 항목

한 학기짜리 프로젝트라 문제를 나중으로 미뤄두면 잊히기 쉽다. `docs/PROGRESS.md` 맨 위에 "미해결 위험" 목록을 별도로 유지하고, 해결되기 전까지 점검 때마다 다시 확인한다.

```markdown
## 미해결 위험
- [ ] pasco 라이브러리가 심박수·폐활량 센서를 공식 지원하지 않음 (§11 세션 F 참고).
      실물 센서로 연결 테스트 필요. 안 되면 §5의 source:"manual" 경로로 대체.
- [ ] (새로 발견되는 위험을 여기에 추가)
```

새 위험을 발견하면(예: 특정 세션이 계속 막힘, 센서 수량 부족 등) 이 목록에 추가한다. 해결되면 체크하고 아래 일반 점검 기록으로 옮긴다.

### 용어·톤 일관성 점검

실험 1·2·3은 서로 다른 세션(다른 모델일 수도 있음)이 독립적으로 작성하므로 버튼 이름, 말투, 단어 선택이 미묘하게 달라질 수 있다. 가끔(전 실험이 화면 골격을 갖춘 뒤) 세 화면의 문구를 훑어보고 `docs/SPEC.md` §14(용어 통일) 기준과 맞는지, 서로 다른 표현을 쓰고 있지 않은지 확인한다. 발견하면 통일 여부를 사용자에게 확인 후, 해당 세션들에 각각 수정 프롬프트를 작성한다.

### 배포 전 점검 (개학 기준일)

- [ ] 실험 1(§10.1)의 화면·분석 5단계가 처음부터 끝까지 끊기지 않고 동작하는가

이 목록이 개학 1주일 전까지 전부 체크되지 않으면 사용자에게 먼저 알린다.

### G는 계속 열어 둔다

G는 한 번 실행하고 끝내는 세션이 아니라 **프로젝트가 끝날 때까지 유지**한다. 새로 세션을 여는 대신 `claude --continue`로 이어가서, 지금까지 점검한 이력을 계속 기억하게 한다.

## 작업 방식 — 토큰·비용 절약

여러 세션이 같은 저장소에서 동시에 작업하는 구조라 문서를 반복해서 읽는 비용이 쉽게 커진다. 아래를 지킨다.

### 1. 읽는 범위를 좁힌다

- `docs/SPEC.md`(37KB)를 매번 통째로 읽지 않는다. 위 "시작하기 전에" 절의 범위만 읽는다.
- 다른 세션이 이미 만든 파일은 **함수 시그니처나 사용법이 필요할 때만** 열어본다. 전체를 훑어보지 않는다.
- 세션 G는 예외로, 점검이 본업이라 여러 파일을 훑어봐야 한다. 다만 한 번에 다 읽지 말고 git diff로 **바뀐 부분 위주**로 확인한다.

### 2. 방금 쓴 내용을 다시 읽지 않는다

- 자신이 방금 작성한 파일을 확인 목적으로 다시 `Read`하지 않는다. 직전에 쓴 내용은 기억하고 있으므로 그 기억으로 이어서 작업한다.
- 코드가 실제로 동작하는지 확인해야 할 때만 실행·렌더링해서 검증한다. 내용 재확인을 위한 재독은 하지 않는다.

### 3. 세션이 끊기면 새로 시작하지 않고 이어간다

- 인터넷이 끊기거나 창을 닫았다면 같은 폴더에서 `claude --continue`로 이어간다.
- 새 세션을 열면 지금까지의 맥락과 이미 읽은 문서를 처음부터 다시 읽어야 한다. 가능하면 피한다.

### 4. 먼저 완성된 실험의 패턴을 재사용한다

- 실험 2·3(세션 C·D)은 실험 1(세션 B)이 먼저 완성된 뒤 시작한다.
- 시작할 때 "`js/exp1.js`를 참고해서 같은 구조로 만들어라"처럼 기존 파일을 기준점으로 제시받는다.
- §12(고민해서 반영하기)는 실험 1에서 이미 검증된 판단은 반복해서 처음부터 재사고하지 않는다. **자기 실험이 실험 1과 다른 부분**(그래프 종류, 질문 내용, 특수 기능)에 판단을 집중한다.

### 5. 작업을 단계로 쪼개서 확인받는다

- "전부 다 만들어라" 한 번에 큰 작업을 맡기지 않는다. 설정(config) → 화면 골격 → 기능 연결 순으로 나누어, 단계마다 방향을 확인받고 진행한다.
- 방향이 틀린 채로 길게 진행하면 전체를 다시 만들어야 하므로, 잘게 쪼개는 편이 총량은 더 적다.

### 6. 모델은 작업 성격에 맞게 쓴다

- 새로운 구조를 설계하는 작업(세션 A, 실험 1의 비계 질문 설계)은 무거운 모델을 쓴다.
- 이미 정해진 패턴을 따라가는 작업(실험 2·3, 교사 대시보드, 수집기)은 가벼운 모델로 충분하다. 사용자가 세션 시작 시 지정한 모델을 임의로 바꾸지 않는다.
- 세션 G는 **기본은 Sonnet**을 쓴다. 다만 아래 상황에서는 `/model`로 잠깐 Opus로 전환한다.
  - 변경 요청이 모호하거나 여러 세션에 걸쳐 있어 담당을 판단하기 어려울 때
  - §12 이행 감사 — 비계 질문이 진짜로 재설계됐는지 판단할 때
  - `docs/SPEC.md`·`CLAUDE.md`를 직접 수정할 때 (모든 세션이 참조하는 문서이므로 실수의 여파가 크다)
  - git 상태 확인, 파일 소유권 체크, 배포 전 체크리스트처럼 기계적인 확인은 Sonnet으로 충분하다.
  - Haiku로는 내리지 않는다. G가 만지는 문서는 다른 모든 세션의 기준이 되므로, 비용을 아끼려다 판단 실수가 전체로 퍼지는 위험이 더 크다.

### 7. 커밋을 자주 남긴다

- 작업 한 덩어리가 끝나면 `git add -A && git commit -m "..."`으로 남긴다.
- 커밋 메시지에 무엇을 했는지 한 줄로 남긴다. 세션 G가 이 메시지로 진행 상황을 파악한다.

## 작업 방식 — 그 외

- 파일 하나가 400줄을 넘으면 사용자에게 알리고 분리를 제안한다
- 주석은 한국어로 단다
- 명세대로 만들었는데 학생이 결론에 이를 수 없는 구조라면, 그것은 명세의 결함이다.
  구현을 강행하지 말고 사용자에게 알린다 (SPEC §12.3)
